C. Lọc Interactions

Các bước lọc tuần tự:
  1. Ép kiểu: user_id, recipe_id → int, rating → numeric, date → datetime
  2. Drop null: Loại dòng thiếu user_id, recipe_id, hoặc rating
  3. Rating range: Chỉ giữ rating ∈ [1, 5] (loại rating=0)
  4. Dedup: Xóa trùng theo (user_id, recipe_id, date, rating), giữ bản cuối

D. Iterative Sparse Filter (quan trọng!)

🔁 Tại sao cần lọc lặp?

Trong recommendation system, user/recipe có quá ít interactions sẽ không đủ tín hiệu để học. Nhưng khi loại user ít, một số recipe cũng mất interactions → rơi xuống dưới ngưỡng → cần lặp lại cho đến khi ổn định.

def iterative_sparse_filter(df, min_u=3, min_i=3): while True: n_before = len(df) # Loại user có < 3 interactions u = df['user_id'].value_counts() df = df[df['user_id'].isin(u[u >= min_u].index)] # Loại recipe có < 3 interactions i = df['recipe_id'].value_counts() df = df[df['recipe_id'].isin(i[i >= min_i].index)] if len(df) == n_before: break # Đã hội tụ return df

C. TF-IDF Vectorization

🧠 TF-IDF là gì?

TF-IDF (Term Frequency–Inverse Document Frequency) biến text thành vector số. Từ nào xuất hiện nhiều trong 1 document nhưng ít trong toàn bộ corpus → có trọng số cao hơn (quan trọng hơn).

TF (Term Frequency)

Từ xuất hiện bao nhiêu lần trong document đó? "sugar" xuất hiện 3 lần → TF cao.

IDF (Inverse Doc Frequency)

Từ xuất hiện trong bao nhiêu documents? "salt" xuất hiện ở 80% recipes → IDF thấp (ít phân biệt).

tfidf_vectorizer = TfidfVectorizer( max_features=20000, # Giữ tối đa 20K từ/cụm từ ngram_range=(1, 2), # Unigram + Bigram ("sugar", "brown_sugar") min_df=5, # Từ phải xuất hiện ≥5 documents max_df=0.95, # Bỏ từ xuất hiện >95% documents ("and", "the") sublinear_tf=True # Dùng 1 + log(TF) thay vì TF thô ) # Output: ma trận sparse (77300 × 20000) recipe_tfidf_matrix = tfidf_vectorizer.fit_transform(df_recipes["combined_text"])

Pending Top-K Cosine Similarity

Tính cosine similarity giữa các recipe TF-IDF vectors. Lưu top-K (vd: 50) neighbors gần nhất cho mỗi recipe → dùng cho Content-Based recommendation.

Pending User Features

avg_rating_given, rating_count, activity_level, favorite_tags (tính từ train only). Fallback mặc định cho cold-start users.

Pending Popularity Features

7. Bước 4: Mô hình Recommendation (Kế hoạch)

Notebook: 04_recommendation_model.ipynb — 5 phương pháp từ đơn giản đến phức tạp.

Baseline Popularity-Based
Ý tưởng

Gợi ý những món phổ biến nhất (rating cao + nhiều lượt đánh giá). Dùng cho cold-start users (user mới, chưa có lịch sử).

Công thức
# Bayesian average để tránh bias từ recipe ít ratings C = global_mean_rating # ~4.41 m = min_ratings_threshold # ví dụ 10 popularity_score = (v/(v+m)) * R + (m/(v+m)) * C # v = số ratings, R = mean rating của recipe
Content Content-Based Filtering
Ý tưởng

User thích recipe A → tìm recipes có nội dung tương tự A (dựa trên nguyên liệu, tags, mô tả).

Cách hoạt động
# 1. Mỗi recipe = 1 TF-IDF vector (20K chiều) # 2. Cosine similarity giữa 2 recipes: similarity(A, B) = (A · B) / (||A|| × ||B||) # Giá trị ∈ [0, 1]: 1 = giống hoàn toàn, 0 = khác hoàn toàn # 3. Với mỗi recipe user thích → lấy top-K neighbors # 4. Gộp + rank → đề xuất
CF Collaborative Filtering
Ý tưởng

Users có sở thích giống nhau → gợi ý món mà user tương tự đã thích. Không cần biết nội dung món.

Matrix Factorization
# Ma trận Rating R (users × recipes) rất sparse (~99.97% empty) # Phân tích thành 2 ma trận nhỏ: R ≈ U × VT # U: user matrix (30K × k) — k latent factors # V: recipe matrix (77K × k) — k latent factors # k thường = 50-200 # Dự đoán rating: r̂(user, recipe) = U[user] · V[recipe]
User-Based CF

Tìm users tương tự → gợi ý món họ thích. Ưu: dễ hiểu. Nhược: chậm khi nhiều users.

Item-Based CF

Tìm items tương tự (theo pattern đánh giá). Ưu: ổn định hơn, cache được. Nhược: cold-start items.

Core Hybrid Recommendation
Kết hợp Content-Based + Collaborative Filtering

Mỗi phương pháp có điểm yếu riêng. Hybrid giải quyết bằng cách kết hợp:

🆕
New User

Chưa có lịch sử → dùng Popularity

📝
Ít lịch sử

Ít ratings → ưu tiên Content-Based

🎯
Nhiều lịch sử

Đủ data → dùng CF + Content

# Weighted hybrid: final_score = α × content_score + β × cf_score + γ × popularity_score # α, β, γ thay đổi tùy context: # Cold-start user: α=0, β=0, γ=1 (chỉ popularity) # Warm user: α=0.4, β=0.5, γ=0.1 # + Constraint filter: minutes ≤ X, calories ≤ Y
Filter Constraint-Based Logic

Lọc kết quả cuối cùng theo ràng buộc người dùng: thời gian nấu tối đa, giới hạn calories, loại trừ nguyên liệu dị ứng, chỉ lấy tag cụ thể (vegetarian, dessert, ...). Đây là post-filter áp dụng sau khi model cho ra danh sách gợi ý.

Mỗi recipe chỉ dùng ~20-50 từ trong vocabulary 20,000 từ → ~99.8% ô = 0. Giống như ma trận rating, hầu hết ô đều trống. Vì vậy dùng scipy.sparse để tiết kiệm bộ nhớ (chỉ lưu ô khác 0).

B. Cosine Similarity — So sánh 2 recipes

Ý tưởng: Đo "góc" giữa 2 vectors

Mỗi recipe là 1 mũi tên (vector) trong không gian 20,000 chiều. Cosine similarity đo góc giữa 2 mũi tên — góc càng nhỏ, 2 recipes càng giống nhau.

Góc giữa 2 vectors (chiếu 2D)
Ứng dụng trong dự án: Top-K Neighbors

Với mỗi recipe, tính cosine similarity với tất cả recipes khác → lấy Top-50 recipes giống nhất (neighbors). Khi user thích Banana Bread → gợi ý các neighbors của nó (có thể là Zucchini Bread, Blueberry Muffins, ...).

C. Ma trận Rating (User × Recipe)

Cách Collaborative Filtering "nhìn" dữ liệu

Tất cả interactions được xếp thành 1 bảng lớn: hàng = user, cột = recipe, ô = rating. Trong dự án: 30,836 users × 77,300 recipes.

Ma trận Rating R (thu nhỏ: 6 users × 6 recipes)
Rating 5 Rating 3-4 Rating 1-2 Chưa đánh giá Cần dự đoán!
Sparsity = 99.97%

30,836 users × 77,300 recipes = 2.38 tỷ ô. Nhưng chỉ có 706,514 ratings → 99.97% ô trống. Đây là thách thức lớn nhất: làm sao điền các ô "?" dựa trên các ô đã biết?

8. Bảng thuật ngữ

Giải thích nhanh các khái niệm quan trọng trong dự án.

Thuật ngữGiải thích
TF-IDFTerm Frequency–Inverse Document Frequency. Cách chuyển text thành số: từ quan trọng (xuất hiện nhiều trong 1 doc, ít trong toàn corpus) có trọng số cao.
Cosine SimilarityĐo độ tương tự giữa 2 vectors bằng góc giữa chúng. Giá trị 0→1. Dùng để so sánh 2 recipes dựa trên TF-IDF vectors.
Sparse MatrixMa trận mà đa số giá trị = 0. Ma trận rating (30K users × 77K recipes) có ~99.97% ô trống. Dùng scipy.sparse để tiết kiệm bộ nhớ.
Matrix FactorizationPhân tích ma trận rating R ≈ U×VT thành 2 ma trận nhỏ hơn. Mỗi user/recipe được biểu diễn bằng k "latent factors".
Cold StartVấn đề khi user hoặc recipe mới chưa có interaction nào → CF không hoạt động. Giải pháp: Popularity hoặc Content-Based.
Data LeakageKhi thông tin từ test/validation set vô tình rò rỉ vào training → model "gian lận". Giải pháp: split trước khi tạo features.
Iterative Sparse FilterLọc lặp đi lặp lại: loại user ít interaction → recipe mất rating → loại tiếp → lặp cho đến khi hội tụ.
Bayesian SmoothingKỹ thuật tránh bias: recipe chỉ có 1 rating = 5.0 không nên rank cao hơn recipe có 100 ratings trung bình 4.8.
Hybrid ModelKết hợp nhiều phương pháp recommendation (Content + CF + Popularity) để bù trừ điểm yếu của nhau.
Latent FactorsCác "đặc trưng ẩn" mà model tự học. Ví dụ: factor 1 có thể đại diện cho "món ngọt", factor 2 cho "món nhanh".
Anti-Leakage SplitSplit train/val theo thời gian (data cũ = train, data mới = val) thay vì random, để mô phỏng thực tế.
Sublinear TFDùng 1 + log(tf) thay vì tf thô. Giảm ảnh hưởng của từ lặp quá nhiều lần trong 1 document.
n-gramCụm n từ liên tiếp. Unigram = 1 từ ("sugar"), Bigram = 2 từ ("brown sugar"). Dự án dùng (1,2)-gram.
min_df / max_dfmin_df=5: bỏ từ xuất hiện <5 docs (quá hiếm). max_df=0.95: bỏ từ xuất hiện >95% docs (quá phổ biến, không phân biệt).
